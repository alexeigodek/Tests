The Pattern Framework

Every MFT requirement a prospect brings can be mapped to one (or a combination) of three fundamental patterns. Understanding which pattern applies is the first question to answer in every POC scoping call.

# Pattern Who Initiates Typical Use Case
1 Internal Orchestration Boomi Runtime Agent (event-driven) Folder_A → Folder_B on legacy ERP
2 MFT as Client Boomi (scheduled pull or push) Pull from S3 every hour, push to partner SFTP
3 MFT as Server External partners (connect inbound) 50 bank vendors upload files to Boomi SFTP endpoint
Each step in this module covers one pattern: the architecture, the demo setup, the discovery questions to ask, and the key proof points to land with the customer.

Pattern 1: Internal Orchestration

Boomi moves files between internal servers automatically using a Runtime Agent — no partner-facing connectivity required.

(Legacy ERP)
Target system

Trigger: New file in Folder A → workflow fires automatically (event-driven or scheduled poll)

What Makes This Pattern Distinct

No partner-facing connectivity — this is purely internal file movement between systems within the same network or VPC
The Runtime Agent polls or listens for new files, then triggers the routing workflow automatically
Supports any protocol the internal systems use: SMB, NFS, local disk, SFTP, FTPS, FTP
Can apply transformation, filtering, renaming, or enrichment rules during routing

Demo Configuration Steps

  1. Deploy a Boomi Atom on a Linux or Windows VM within the customer's network (or use an existing Atom if one is deployed). Verify the Atom is Online in the Boomi platform.

  2. Create a watched directory — designate a local folder (Folder A) on the Atom's host machine as the source. Configure the Boomi MFT Runtime to watch this directory for new file arrivals.

  3. Create the destination — configure Folder B on the same or different internal server. This simulates the legacy ERP landing zone.

  4. Build the routing workflow — in the MFT platform, create a workflow rule: When new file arrives in Folder A → move/copy to Folder B. Add any filename filters or transformations needed.

  5. Test end-to-end — drop a test file into Folder A. Verify it appears in Folder B within the configured poll interval. Check the Activity Log to confirm the full audit trail.

Discovery Questions

"Are there any files today that move manually between internal servers — someone copying files from one system to another on a schedule?"
"Do you have legacy systems that can only consume files from a specific local directory path rather than an API?"
"How do you currently handle file routing failures — what's the error notification and retry story?"
POC Checklist items for this pattern: Deploy Atom → Verify Online → Configure watch directory → Build Folder_A → Folder_B workflow → Drop test file → Confirm delivery in Activity Log

Pattern 2: MFT as Client

Boomi initiates every connection — pulling from external sources or pushing to partner endpoints on a schedule. External systems never call Boomi.

→ SFTP Push →
🔒 Partner
SFTP Server
External target

Boomi holds all credentials and initiates every session. External systems never need to call inbound to Boomi.

What Makes This Pattern Distinct

Boomi holds all credentials centrally — external systems never need firewall rules allowing inbound connections to Boomi
Supports multiple target protocols per transfer: SFTP, FTPS, FTP, AWS S3, Azure Blob, Google Cloud Storage — all configurable per connection
Built-in retry logic and transfer-state tracking — if a pull fails, Boomi retries without manual intervention
Scheduling is flexible: fixed intervals (every hour), CRON expressions, or event-triggered

Demo Configuration Steps

  1. Configure an External Endpoint — in the MFT platform, create a connection to the external source (e.g., an AWS S3 bucket with access key/secret, or a partner SFTP server with SSH key). Store credentials securely in the platform.

  2. Create a Pull transfer definition — define what to pull: which bucket/path, file filter (e.g., *.csv), and what to do with the file after retrieval (delete, archive, or leave in place).

  3. Set the schedule — configure the transfer to run every hour (or on a custom CRON). Trigger a manual run to verify connectivity.

  4. Verify retrieval — confirm files appear in the configured landing zone and check the Activity Log for transfer timestamps, byte counts, and status.

  5. (Optional) Configure a Push — add a second step: after pulling from S3, push the file to a partner SFTP server. This demonstrates the full pull-process-push pipeline in one workflow.

Discovery Questions

"Do any of your current integrations involve scheduled jobs that connect to external FTP servers to pick up files? Who manages those scripts today?"
"How do you handle the case where a partner's SFTP server is temporarily unavailable? What's your current retry strategy?"
"Do you need to pull from multiple cloud storage providers — S3, Azure Blob, GCS — in the same environment?"
Key differentiator to land: Boomi as client means zero inbound firewall changes for the customer. Their security team will appreciate that no new ports need to be opened on their perimeter.

Boomi hosts the SFTP endpoint. External partners connect and upload directly — each one isolated in their own Virtual Folder.

Auth: SSH Key or Password per partner account. Each partner sees only their own namespace — full multi-tenant isolation.

What Makes This Pattern Distinct

Boomi presents a single public SFTP endpoint — partners connect exactly like any standard SFTP client (FileZilla, WinSCP, custom script)
Virtual Folders enforce multi-tenant isolation — Vendor 1 cannot see or access Vendor 2's files under any circumstance
Full audit trail for every upload: Received → Validated → Processed → Delivered
Scales to 50, 500, or 5,000 partners — each with their own credentials, permissions, and quota settings

Demo Configuration Steps

  1. Create an MFT User account — in the Boomi platform, create a user account for the partner (e.g., "BankVendor1"). Set the authentication method: SSH key preferred, password as fallback for demo.

  2. Create a Virtual Folder — assign the user a Virtual Folder with a defined root path. This is the only directory the partner will be able to see when they connect.

  3. Connect via SFTP client — open FileZilla or WinSCP, connect to the Boomi SFTP endpoint (host, port 22, credentials). Verify the partner can see only their Virtual Folder.

  4. Upload a test file — upload a test file from the client. Switch to the MFT Activity Log and confirm the file is visible with status Received.

  5. Verify routing — confirm the file progresses through Validated → Processed → Delivered in the audit trail. Show the customer this full lineage in a single view.

  6. (Bonus demo) — create a second user and Virtual Folder, connect as that user, and show that they cannot see the first partner's files. This is the multi-tenant isolation proof point.

Discovery Questions

"How many external partners currently send files to you? Are they all connecting to the same endpoint or different ones?"
"How do you provision access today when a new partner needs to start sending files? How long does that take?"
"Have you ever had a situation where a partner uploaded to the wrong location and caused a data issue?"
The money slide: The four-state audit trail — Received, Validated, Processed, Delivered — is the proof point that resonates most with operations teams. Show it during every demo.
