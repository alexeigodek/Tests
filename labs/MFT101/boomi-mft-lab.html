Introduction

Overview

Welcome to Boomi Managed File Transfer (MFT) — your solution for simple, secure, and reliable file transfers. As part of Boomi's unified data integration platform, Boomi MFT consolidates all your file transfer operations into a single platform, streamlining workflows and reducing costs.

If you're managing a fragmented landscape of legacy SFTP servers, custom scripts, and point-to-point solutions, you know how difficult it is to monitor, govern, and scale. Boomi MFT eliminates that complexity — delivering enterprise-grade security, governance, and control in one platform accessible to both technical and business users.

🔒 Enterprise Security

End-to-end encryption, PGP/SSH key management, FTPS certificates, and built-in AV scanning — included, not bolted on.

🔍 Full Visibility

Every transfer is tracked end-to-end. See when a file arrived, was renamed, compressed, delivered — or failed.

🌐 Flexible Deployment

Cloud, on-premises, and hybrid. MFT Runtimes bridge firewalls and access local file systems the cloud can't reach.

🤝 Multi-Party Ready

One flow, many subscribers. Add or remove partners without rebuilding your entire transfer process.

Core Concepts

Foundations

Boomi MFT is built around two applications and a handful of key objects. Getting these clear before you build will save you a lot of time.

Two Applications

⚡ Automated File Transfer

System-to-system. Scheduled batch processes or event-driven transfers. Built around Flows — the main focus of this lab.

📤 File Sharing

User-centric, ad-hoc transfers. Designed for large or sensitive files that people need to send manually.

Key Objects

ObjectWhat it isThink of it as…
OrganisationA participant — internal entity or external partner/supplierA company or department
EndpointConnection credentials and protocol for an organisation (SFTP, S3, etc.)A mailbox or loading dock
FlowThe file transport route. Organisations subscribe as source or target. Flows, not folders.A shipping lane
RuntimeA lightweight agent installed on-premises to reach behind firewalls and local file systemsA local delivery driver
SubscriberAn organisation participating in a flow — as source, target, or bothA channel subscriber
📌
Flows are pub/sub One flow can have multiple sources and multiple targets. You add or remove subscribers without changing anything else. A single organisation can be a source in one flow and a target in another.

Organisations & Endpoints

Platform Setup

Before building a flow, you create the participants and define how they connect. Organisations are the containers; Endpoints hold the connection credentials and protocol type.

Organisations — create or select an existing organisation

Creating an Organisation

1

Navigate to Organisations

Select Organisations in the left navigation. You'll see all existing participants.

2

Add Organisation

Click Add Organisation, give it a meaningful name ("Supplier A", "Headquarters"), and save. This creates the container that holds their endpoints and keys.

3

Add Endpoints

Inside the organisation, click Add Endpoint. Name it, choose the protocol type from the list, and fill in the connection details. An organisation can have multiple endpoints.

4

Test & Validate

Use Test Connection and Validate buttons to confirm connectivity before using the endpoint in a flow. Fix any issues before it goes live.

Organisation Detail — multiple endpoint types on a single organisation

Supported Endpoint Types

SFTP – External Server SFTP – Internal (Hosted) Amazon S3 Azure Blob Storage FTPS FTP Google Cloud Storage Network File Share (SMB) Local File System Boomi File Share
Endpoint types — selecting a protocol when adding a new endpoint
♻️
Endpoints are reusable authentication containers Credentials live in the endpoint. Transfer-specific parameters (paths, schedules, filters) are set when you add the endpoint to a flow. The same endpoint can be a source in one flow and a target in another.

Keys & Certificates

Security

All cryptographic material is managed in one place — no more tracking key files in shared drives or updating scripts when keys rotate.

Key Management — PGP and SSH key sets per organisation
Key TypeUsed ForCreate or Import?
PGP Key SetsEncrypting/decrypting file content in transit and at restCreate (algorithm & key strength) or Import
SSH Key SetsSFTP authentication — share your public key with partners, or import theirsCreate (algorithm & key strength) or Import
CertificatesFTPS connections (TLS mutual auth)Import only — issued externally
⚠️
Watch your key expiry dates The platform clearly shows which keys have expired. Make key rotation a regular operations task — expired keys on a partner endpoint will cause transfers to fail at the worst possible time.

Keys are scoped to an organisation. Multi-tenant by design — one partner's keys are never visible to another, enabling true data sovereignty for all participants.

Runtimes

Hybrid Deployment

A Runtime is a lightweight agent you install on any machine — on-premises, in a VM, or in a private cloud. It extends Boomi MFT behind firewalls and enables transfers to local file systems, network shares, and mapped drives that the cloud cannot reach directly.

Runtime Manager — live runtimes reporting heartbeat to the cloud

Installing a Runtime

1

Create in the Portal

Navigate to Runtimes. Click Add Runtime, give it a name, save — then click the Download button to get the installer.

2

Install via Script

Windows: Copy the generated PowerShell script, paste into an elevated prompt — it installs and self-configures.
Linux: Grab the apt/yum or systemd script, paste into a terminal — same result.

3

Verify in the Portal

Once running, the runtime appears as Alive with its hostname, IP, and last heartbeat timestamp. The cloud manages scheduling and job assignment from here.

🔄
Cloud-orchestrated, locally-executed The runtime polls the cloud for jobs, processes them locally, and reports results back. This means no inbound firewall rules are needed — only outbound HTTPS. The runtime startup sequence is: check for update → validate API token → start processing jobs in a loop.

What a Runtime Can Access

If the operating system can mount or see a path, the runtime can use it:

Local File System (any path) Windows Mapped Drives Network Shares (SMB/NFS) Linux Soft/Hard Mounts Cloud Storage (via credentials)

Heartbeat Alerts

Click any runtime to configure alert recipients for when it goes offline or comes back online. Assign to specific email addresses rather than all admins — keep the signal relevant.

Creating Flows

Core Feature

A Flow is the core unit of Boomi MFT. It defines a file transport route and the organisations that participate in it as sources and targets.

📌
Flows, not folders Traditional MFT is folder-based — logic lives in naming conventions and scripts. Boomi MFT is flow-based — all the logic, routing, and transformation is configured in the flow itself.
Flows — listing of all organisation flows with status, codes, and subscriber counts

Building a Flow

1

Create the Flow

Click Add Flow, give it a descriptive name ("Supplier Orders to HQ"), and save. Additional options: export to a file, export to FileZilla, disable, or delete from the flows list.

2

Add Subscribers

Open Subscribers. All organisations in your instance are shown. Select the ones participating in this flow — Headquarters, Partner 1, Supplier, Vendor — and click Subscribe.

3

Configure Sources

For each source organisation, choose which of their endpoints to collect from — their external SFTP, their S3 bucket, or your internal hosted server. Multiple endpoints per organisation are supported.

4

Configure Targets

For each target, choose the delivery endpoint and path. Define processing, transformations, and outbound triggers.

5

Publish

Push and publish the flow. It becomes active immediately — runtimes and cloud processors start acting on it based on their configured schedules.

Subscribers — all organisations in the instance, shown when adding to a flow

Source Configuration

Flow Detail

When you add a source endpoint to a flow, you configure how Boomi MFT collects files from it. These are per-source settings, separate from the endpoint's base credentials.

Source Configuration — the Flow Endpoint Editor with tabs for each setting category

Processing

Choose whether this source is processed by the Cloud or a specific Runtime. Cloud handles most external connections; runtime is required for on-premises paths or local file systems.

Filtering Options

FilterDescription
Include (Regex)Only collect files matching this pattern — e.g. .*\.csv$ to pull only CSV files
Exclude (Regex)Skip files matching this pattern — e.g. exclude temp or archive files
Date FilterOnly collect files before/after/between specific dates. Critical for servers with years of historical files you don't want to pull accidentally.

Post-Collection Actions

Delete from Source

After successful collection, delete the file. Standard for "pick up once" workflows.

Archive on Source

Move to an archive path instead of deleting. Choose one or the other — not both.

Include Subfolders

Recursively collect from subdirectories under the configured base path.

Encryption & Decryption

If incoming files are PGP-encrypted, configure decryption with a stored private key. Optionally verify authenticity with a signature key. This happens automatically in the processing pipeline.

Processing Options tab — encryption key selection, signing key, and error handling
🛡
AV scanning is automatic Files passing through the Boomi MFT cloud are automatically scanned for viruses and malware. No configuration required — it's on by default.

Trigger Options — When Is a File "Ready"?

Trigger File Settings — control when MFT considers a file ready to collect
Trigger TypeHow It Works
DisabledCollect immediately — no trigger mechanism
Temp FilenameWait for a .tmp or .part suffix to disappear — indicates the file has finished writing
Trigger File (per file)A separate signal file per data file indicates it's ready to transfer
One Trigger FileA single semaphore file indicates the entire batch is ready — deleted after all files transfer

File Renaming

The rename engine is a visual pattern builder. Define matching rules (regex), then construct new filenames using building blocks: original name, timestamp (with format options), static text, and separators. Preview the result live before saving.

Rename Builder — pattern builder with live test string preview

Source Alerts

Configure per-recipient email alerts at the source level:

Alerts Configuration — inactivity and error alerts with per-recipient filtering

Each alert recipient can be filtered to receive only the alert types they care about — success emails, errors only, quarantine notices. Different people, different alert types, same flow.

Target Configuration

Flow Detail

The target side mirrors the source in structure, but defines where and how files are delivered.

Target Configuration — delivery path, concurrency, and outbound options

Key Target Settings

SettingDescription
Delivery PathBase path + optional sub-path for this flow's files. For runtime targets, the runtime must have OS-level access to this path.
ConcurrencyHow many files to deliver simultaneously to this target
Overwrite / Auto-timestampWhat to do if a file with the same name already exists at the target
Outbound Trigger FileGenerate a signal file when delivery is complete, so downstream systems know the batch is ready
Temp FilenameWrite with a temp name while delivering; rename to final only when complete — prevents downstream systems consuming a partial file

Source-to-Target Mapping

The mapping layer lets you route specific sources to specific target paths within the same flow. A catch-all routes everything to one path; specific rules let you send Partner 1's files to Path A and Supplier's files to Path B — all in one flow, no custom code.

🗺
One flow, many routing rules 5 sources, 3 targets, and a mapping table that routes each source's files to the right target paths. All configured in the flow — no scripting required.

Real-World Scenario

Manufacturing Example
Use Case

Manufacturing Supply Chain — Automated Purchase Orders

A manufacturing enterprise manages multiple suppliers for raw materials. When inventory drops below a threshold, the ERP system generates purchase orders. Those orders are transformed by the Boomi integration layer, handed to Boomi MFT, and routed to each supplier simultaneously — with Headquarters getting an archive copy of everything.

System Architecture

ERP System
Inventory drops → PO generated
→
Boomi Integration
Transforms & routes to MFT API
→
Boomi MFT
Distributes to all subscribers
Supplier A
S3 Bucket
Supplier B
External SFTP
Supplier C
Internal Drop
HQ Archive
SFTP Record

The flow runs on a schedule. Each supplier only sees their own file. Headquarters receives all files for audit.

Step-by-Step Walkthrough

1

Boomi Integration Drops Files into MFT

The integration process generates PO files and calls the MFT API — passing the storage key, secret, and target flow/endpoint — to stage the files for distribution. One file per supplier, named accordingly ([001]_110326.xml, [002]_110326.xml, etc.).

2

MFT Receives and Stages the Files

MFT picks up the files from the "Boomi Drop Off" flow endpoint. The Activity view immediately shows the files arriving — which flow, which organisation, file size, and timestamp.

3

Flow Routing Distributes to Each Subscriber

Using the flow's mapping rules: [001] file → Partner 1, [002] → Partner 2, [003] → Partner 3. Headquarters receives a copy of all files. Each supplier only sees their own PO — no cross-tenant visibility.

4

Failures Are Caught and Alerted Immediately

If Partner 3's S3 endpoint is down, MFT logs the failure in the Activity view and fires an alert email. The other deliveries continue normally. The broken endpoint shows a delivery count of "0/1" — easy to spot.

Activity View — live file routing showing multiple flows, partners, and delivery status
Activity view — per-partner file routing with delivery status across all subscribers

Boomi Integration ↔ Boomi MFT

The connection between Boomi Integration and Boomi MFT is bidirectional and API-driven. The integration process uses the MFT API connector with stored credentials (storage key + secret) to identify the target flow endpoint. The reverse works too — MFT can stage a file for Boomi Integration to pick up, triggering downstream processing. Both directions are visible and traceable in the Activity view.

Boomi Integration — the MFT connector in action, pushing files to a flow endpoint
🔌
API-first throughout Everything in the MFT UI — create flows, manage organisations, trigger transfers, view activity — is also available via API. Your integration processes, ITSM tools, and CI/CD pipelines can fully automate MFT management.

Monitoring & Alerts

Operations

Boomi MFT responds to failures immediately — no polling, no waiting for someone to check a dashboard.

The Alerts Page

Alerts — system-generated alerts showing connection errors, inactivity, and quarantined files

The Alerts page shows all active alerts across every flow and organisation — severity (Critical / Warning), the affected flow and partner, and the alert type. Once an issue is resolved, clear the alert to keep the view clean. Alerts are also emailed to configured recipients as they fire.

Alert Types

TypeWhat It Means
Connection ErrorMFT can't reach an endpoint — credentials expired, server down, or firewall change
InactivityNo files transferred in the expected window — partner may have a problem on their side
QuarantinedFile failed AV scan or couldn't be decrypted — held for manual review
Path ErrorConfigured path no longer exists or accessible — common after infrastructure changes

Activity View — Full File Lifecycle

Every file in the Activity view shows its complete lifecycle: received → renamed/compressed/encrypted → delivered (or failed) — with timestamps at each step. Filterable by flow, organisation, endpoint, filename, date range, and transfer status. Exportable for compliance reporting.

🔔
Downstream notifications MFT doesn't just notify people — it can notify downstream systems. If a delivery fails, integration processes waiting for that file can be told immediately, enabling automated retry or escalation logic.

Administration

Platform Management

User Roles

RoleAccess Level
AdministratorFull access — users, organisations, flows, runtimes, keys, configuration, and audit log
Instance UserManages flows and organisations, but not system-wide settings
Organisation UserScoped to a single organisation — manages their endpoints, sees only their transfers
Transfer UserSelf-service portal access for ad-hoc file sharing — can't see other users' files
🔐
True multi-tenant isolation Organisation Users have no visibility into other organisations or their transfers. Transfer Users see only their own files. Partners get self-service access without any cross-tenant exposure risk.

Audit Log

Admin › Audit Log — every action tracked by user, event type, and timestamp

Every administrative action is logged — configuration changes, user creation, flow updates, key imports, alert clears. Who did what, and when. Exportable to CSV. This applies to everything including API-driven changes, so your CI/CD pipelines and automation are also fully auditable.
