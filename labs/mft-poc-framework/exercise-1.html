Why This Matters in a POC

Security is almost always the first objection in an MFT deal. Prospects in banking, healthcare, and logistics will ask: "How is the file secured in transit? What auth methods do you support? Where does the data live?"

This module gives you authoritative, technically accurate answers — and the vocabulary to match the customer's security team.

The Key Differentiators for Customer Conversations

FTP — The baseline, never recommended

The original standard. Data and credentials travel in plaintext. Port 21 for control, a dynamically negotiated port for data. Never use FTP for sensitive data. If a prospect is currently using FTP, that's an immediate risk conversation you can lead with.

FTPS — Certificate-based, common in regulated industries

FTP layered with TLS/SSL encryption using X.509 certificates. Supports two modes: Explicit (starts unencrypted on port 21, upgrades via STARTTLS) and Implicit (always encrypted on port 990). Commonly required in banking and healthcare for certificate-based mutual authentication. Firewall-unfriendly due to multiple port usage.

SFTP — The gold standard for external partner exchange

Not related to FTP — it runs entirely over SSH on a single port (22). No separate data channel means simpler firewall rules. Uses SSH key pairs for the most secure authentication. The recommended default for all external partner connectivity in Boomi MFT.

SE Tip: When a prospect asks "which protocol should we use?", default to SFTP unless they have a compliance requirement mandating X.509 certificates (in which case FTPS). Never recommend plain FTP for production use.
Step 2 of 3

Authentication Methods

Three authentication approaches — know when to recommend each and how to position them against a prospect's security requirements.

Password-Based Authentication

Username and password credentials. Simple to set up, but vulnerable to brute force, credential stuffing, and insider threats. Least preferred for MFT deployments. Acceptable only in low-risk, internal scenarios with compensating controls (VPN, IP allowlist). If a customer is using this today, it's a security upgrade conversation.

SSH Key-Based Authentication (SFTP)

Uses asymmetric cryptography — a public/private key pair. The partner holds the private key; Boomi MFT stores the public key. Authentication happens without ever transmitting a password. The gold standard for SFTP connections. Eliminates password rotation burden and is phishing-resistant. Position this for all external partner connections.

Certificate-Based Authentication — X.509 (FTPS / AS2)

Used with FTPS and AS2. Both parties exchange digital certificates signed by a Certificate Authority (CA). Provides mutual authentication — each side verifies the other's identity. Highest assurance level and often mandated by compliance frameworks (PCI-DSS, HIPAA, EDI/AS2 trading partner agreements). Requires certificate lifecycle management — a complexity worth calling out.

Quick Reference: When to Recommend What

Scenario Recommended Auth Why
External partner SFTP exchange SSH Key Pairs No passwords to rotate, phishing-resistant
Banking / regulated FTPS requirement X.509 Certificates Compliance mandate, mutual auth
EDI / AS2 trading partners X.509 Certificates AS2 spec requires cert-based signing
Internal low-risk transfer Password (with VPN) Acceptable only with network controls
During the Mock POC checklist (Module 3), you'll be required to replace password auth with SSH key auth and verify the password no longer works. Practice the transition now if you haven't already.

Encryption & Architecture

How Boomi MFT secures data at rest, in transit, and across a DMZ boundary — and how to explain it to a security-conscious prospect.

The Three Security Layers

At Rest

Data is encrypted while stored on the MFT server or disk. Files sitting in queues, virtual folders, or audit logs are encrypted using AES-256. Key question to address: "Where does the encrypted data reside — on your infrastructure or Boomi's?"

In Transit

TLS/SSL secures the data channel during active file transfer. SFTP uses SSH transport; FTPS uses TLS over the data channel. Data is never sent as plaintext between endpoints. Ask the prospect: "Do your trading partners support TLS 1.2 or higher?"

DMZ & Proxy

MFT acts as a buffer between the public internet and the secure internal network. External partners connect to the DMZ endpoint; internal systems never have direct internet exposure. Critical for financial services and healthcare compliance.

DMZ Architecture Explained

The DMZ (Demilitarized Zone) pattern is a common requirement for enterprise MFT deployments. Here's how to describe it to a prospect's security team:

External Partners
(Internet)
→ SFTP/FTPS →
DMZ
Boomi MFT Endpoint
→ Internal only →
Secure Network
ERP / Internal Systems

Partners never reach the internal network directly. The MFT server in the DMZ receives files, validates them, then routes inward through an internal-only connection.

Discovery question to ask: "Do you currently expose any internal servers directly to the internet for file transfer? If so, how are you managing that risk?" This surfaces DMZ requirements and positions Boomi as the cleaner architecture.
